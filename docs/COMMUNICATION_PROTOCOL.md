# PLC与Modbus通信协议

## 1. 文档范围

项目同时实现两条独立的PLC模拟通信链路：

1. 基于TCP的UTF-8文本协议，用于直观演示心跳、请求响应和检测握手。
2. 标准Modbus TCP协议，用线圈和保持寄存器表达检测状态与结果。

两条链路目前使用本机模拟服务端验证。接入真实PLC时，应根据PLC品牌、地址表示方式、扫描周期和现场网络重新确认映射与超时参数。

## 2. 默认连接参数

| 项目 | 文本协议 | Modbus TCP |
|---|---:|---:|
| 默认地址 | `127.0.0.1` | `127.0.0.1` |
| 默认端口 | `1502` | `1503` |
| 传输层 | TCP | TCP |
| 上位机角色 | 客户端 | 客户端 |
| 模拟PLC角色 | 服务端 | 服务端 |

系统设置保存文本协议端口，Modbus TCP端口按“文本端口+1”计算，因此文本端口最大允许值为65534。

## 3. TCP文本协议

### 3.1 报文格式

- 字符编码：UTF-8
- 报文结束符：换行符`\n`
- 最大请求长度：256个字符
- 命令名称不区分大小写
- 同一连接上的普通请求按顺序发送，避免响应错配

### 3.2 基础命令

| 发送方 | 请求 | 正常响应 | 用途 |
|---|---|---|---|
| 上位机 | `PING` | `PONG` | 人工连通性测试 |
| 上位机 | `HEARTBEAT` | `ALIVE` | 自动心跳检测 |

人工PING默认等待2秒，连接默认等待3秒。心跳使用独立周期任务检测链路，失败后关闭当前连接并进入自动重连流程。

### 3.3 检测握手

`cycleId`是本次检测周期标识，用于防止上一周期的迟到报文污染当前结果。

```text
模拟PLC  → 上位机：START <cycleId>
上位机   → 模拟PLC：BUSY <cycleId>
模拟PLC  → 上位机：ACK BUSY <cycleId>
上位机执行视觉检测
上位机   → 模拟PLC：RESULT <cycleId> OK
                          或 RESULT <cycleId> NG
模拟PLC  → 上位机：ACK RESULT <cycleId>
```

完整示例：

```text
START 1024
BUSY 1024
ACK BUSY 1024
RESULT 1024 OK
ACK RESULT 1024
```

只有自动模式允许PLC触发检测。握手执行期间，状态机从`Ready`进入`Running`，检测完成后进入`Completed`。

### 3.4 错误响应

| 响应 | 含义 |
|---|---|
| `ERR EMPTY_COMMAND` | 请求为空 |
| `ERR UNKNOWN_COMMAND` | 命令名称或参数数量无法识别 |
| `ERR MESSAGE_TOO_LONG` | 请求超过256个字符 |
| `ERR CYCLE_MISMATCH` | 周期号与当前活动周期不一致 |
| `ERR INVALID_RESULT` | RESULT结果不是OK或NG |

### 3.5 重连策略

连接失败后按照1秒、2秒、4秒、5秒的退避间隔继续重试，后续间隔最大保持5秒。人工断开会取消自动重连，程序退出时通过取消令牌停止后台任务。

## 4. Modbus TCP协议

### 4.1 协议参数

- MBAP协议标识：`0`
- 单元标识：`1`
- 轮询间隔：默认200 ms
- 单次自动握手读写超时：1秒
- 模拟数据区：100个线圈、100个保持寄存器

当前实现支持以下功能码：

| 功能码 | 名称 | 用途 |
|---:|---|---|
| `0x01` | Read Coils | 读取检测握手线圈 |
| `0x03` | Read Holding Registers | 读取周期号和结果代码 |
| `0x05` | Write Single Coil | 写入单个握手状态 |
| `0x06` | Write Single Register | 写入周期号或结果代码 |

不支持的功能码返回异常码`0x01`；地址越界返回`0x02`；非法参数返回`0x03`；服务端内部处理失败返回`0x04`。

### 4.2 线圈映射

代码内部使用从0开始的协议地址。部分PLC软件会显示为`00001`起始编号，接入真实设备时必须确认是否存在1位偏移。

| 协议地址 | 名称 | 写入方 | 含义 |
|---:|---|---|---|
| `0` | `START` | PLC | 置1请求开始检测；上位机收到后清0 |
| `1` | `BUSY` | 上位机 | 置1表示正在检测，完成后清0 |
| `2` | `DONE` | 上位机 | 置1表示本周期完成 |
| `3` | `OK` | 上位机 | 置1表示产品合格 |
| `4` | `NG` | 上位机 | 置1表示产品不合格或检测异常 |

### 4.3 保持寄存器映射

| 协议地址 | 名称 | 写入方 | 数值含义 |
|---:|---|---|---|
| `0` | `CycleId` | PLC | 0～65535的检测周期号 |
| `1` | `ResultCode` | 上位机 | `0`无结果、`1`为OK、`2`为NG |

### 4.4 Modbus检测时序

```text
PLC：写 CycleId
PLC：START=1
上位机轮询发现 START=1
上位机：ResultCode=0，DONE=0，OK=0，NG=0
上位机：BUSY=1，START=0
上位机执行视觉检测
上位机：ResultCode=1或2
上位机：OK/NG写入对应结果
上位机：BUSY=0，DONE=1
PLC读取结果并进入下一周期
```

在当前模拟流程中，下一次触发前会重置握手区。真实PLC程序还应明确DONE确认和复位条件，避免PLC长时间保留DONE导致重复读取。

## 5. 异常与并发原则

- TCP与Modbus请求分别使用异步互斥，保证同一连接只有一个请求等待响应。
- 所有网络操作均设置超时，禁止永久等待设备。
- 断线后释放旧套接字和网络流，再创建新连接。
- 状态机与运行模式策略会拒绝非法状态、手动模式下的远程触发和重复检测。
- 相机、通信或检测异常会记录运行日志并进入报警/故障处理链路，而不是让WPF进程直接退出。

## 6. 真实PLC接入检查表

1. 确认PLC品牌、CPU型号、固件和支持的协议。
2. 确认PLC端是客户端还是服务端，以及实际IP和端口。
3. 确认Modbus地址是0基还是1基显示。
4. 确认线圈、寄存器、字节序和结果代码映射。
5. 确认PLC扫描周期、上位机轮询周期和允许检测节拍。
6. 明确START、BUSY、DONE、结果确认和复位的责任方。
7. 在拔网线、PLC停机、上位机重启和报文超时场景下验证恢复。
8. 记录真实网络往返、连续运行时间和恢复次数，重新建立现场基线。
